API Reference

HTTP APIs available to agents and the Proof of Reserves / Token Explorer frontends. All endpoints return JSON. CORS is enabled for *.

If you're an agent, start here (MCP): Connect the MCP server (npm run mcp) for typed tools and discovery—no HTTP wiring. Or use GET /api/openapi.json for the full schema.

Machine-readable schema: GET /api/openapi.json — OpenAPI 3.0 spec (endpoints, request/response schemas, error codes). Use this for agent tooling and codegen.

For agents: two main actions

  1. Swap (SOL → BTC): GET /api/swap/min and GET /api/swap/estimate, then POST /api/swap/create. You get the tx id and can disconnect; poll GET /api/swap/status/:id only if you need completion status.
  2. Create your own token: POST /api/invoices with token details → receive invoice (fee in SOL) → pay to treasury → POST /api/invoices/confirm with invoice_id and tx_signature → we create the SPL token and add it to the listing.

Agent quickstart (linear path)

Fastest way for an agent to succeed:

  1. Discover: GET /api/openapi.json (or connect the MCP server with npm run mcp) to get endpoints and schemas.
  2. Choose flow: Swap SOL→BTC or create token (see “For agents: two main actions” above).
  3. Execute: Follow the deterministic steps for that flow (tables and curl examples below).
  4. Status: Poll the status endpoint (/api/swap/status/:id or /api/invoices/:id) only if you need completion; otherwise you can disconnect after receiving the tx id or invoice.

Deterministic flows (step-by-step)

Agents should follow these state transitions; poll status endpoints instead of guessing.

Flow 1: Token creation

StepAction
1POST /api/invoices — body: { "name", "symbol", "creator_address" } (optional: decimals, supply, description, revoke_*).
2Response: status: "pending_payment", invoice_id, address (treasury), amount (SOL), currency: "SOL".
3Send amount SOL to address (user's wallet signs the tx).
4POST /api/invoices/confirm — body: { "invoice_id", "tx_signature" }. Idempotent: already completed returns same success.
5Response: status: "completed", token_id, mint_address. Optional: GET /api/tokens/{token_id}.
# Step 1 – create invoice
curl -s -X POST https://www.solanaagent.app/api/invoices \
  -H "Content-Type: application/json" \
  -d '{"name":"My Token","symbol":"MTK","creator_address":"YOUR_SOL_ADDRESS"}'

# Step 4 – confirm (after paying amount SOL to address)
curl -s -X POST https://www.solanaagent.app/api/invoices/confirm \
  -H "Content-Type: application/json" \
  -d '{"invoice_id":"UUID-FROM-STEP-1","tx_signature":"SOLANA_TX_SIG"}'

Flow 2: Swap SOL → BTC

StepAction
1GET /api/swap/min — returns minAmountSol, balanceSol.
2GET /api/swap/estimate?amountSol=X — returns estimatedBtcSats.
3POST /api/swap/create — body: { "amountSol": X }. Returns immediately with initiated: true, id (Solana tx signature).
4(Optional) GET /api/swap/status/{id} — status: waiting | finished | failed.
# Steps 1–2
curl -s "https://www.solanaagent.app/api/swap/min"
curl -s "https://www.solanaagent.app/api/swap/estimate?amountSol=0.05"

# Step 3
curl -s -X POST https://www.solanaagent.app/api/swap/create \
  -H "Content-Type: application/json" \
  -d '{"amountSol":0.05}'

Errors (machine-usable)

All errors return JSON: { "error_code", "error", "action?" }. Example: { "error_code": "INSUFFICIENT_BALANCE", "error": "Insufficient SOL balance (including fee reserve)", "action": "top_up_wallet" }.

Safe retry guidance

  • POST /api/invoices/confirm — Safe to retry. If the invoice is already completed, you get the same success response (idempotent). Use this on timeouts or network errors.
  • POST /api/invoices — Do not retry with the same payload expecting the same invoice. Each call creates a new invoice. If you need to retry, use the new invoice_id and pay that invoice.
  • POST /api/swap/create — Not idempotent; each call initiates a new swap. Only retry if the previous request failed (e.g. 502) and you did not receive an id.

1. Base URL

When calling from a browser on this site, use relative paths (e.g. /api/reserves). From an external agent or script, use the full origin (e.g. https://www.solanaagent.app/api/reserves).

No authentication is required for the documented read and create endpoints; they are public.

1.1 Rate limits and constraints

Agents should respect these boundaries:

  • Swap (LI.FI): No API key required; LI.FI allows ~200 requests per 2 hours per client. Optional key at li.fi for higher limits. Minimum SOL amount: use GET /api/swap/min (typically 0.001 SOL); maximum is effectively the reserve balance.
  • Token creation (invoices): No per-agent limit; fee is computed per invoice (estimated on-chain cost + margin). Decimals 0–9; supply is a string (raw units).
  • BTC settlement time: After POST /api/swap/create, Bitcoin can take several minutes up to a few hours; poll GET /api/swap/status/:id for finished or failed.

2. Reserves & proof

GET /api/reserves

Returns Bitcoin and Solana reserve addresses and balances.

Request: No parameters.

Response (200):

curl "https://www.solanaagent.app/api/reserves"

GET /api/proof

Returns a dated message and Bitcoin/Solana signatures proving ownership of the reserve keys.

Response (200): message, timestamp, bitcoin.address, bitcoin.signature, solana.address, solana.signature.

3. Transactions

GET /api/transactions/bitcoin

Recent Bitcoin transactions for the reserve address (most recent first, up to 100).

Response (200): { "transactions": [ { "txid", "blockTime", "blockHeight" }, ... ] }

GET /api/transactions/solana

Recent Solana transactions for the reserve address.

Response (200): { "transactions": [ { "signature", "blockTime", "err" }, ... ] }

4. ABSR (Arbitrage / reserve summary)

ABSR (Agent Bitcoin Strategic Reserve) is 1:1 with the Bitcoin reserve: one satoshi in the reserve = one ABSR. A daily job mints ABSR when the reserve balance exceeds the on-chain supply.

GET /api/arbitrage/summary

BTC/SOL prices (Hyperliquid) and ABSR metrics: absrSupplySats (reserve balance in sats), absrSupplyUsd, absrTokenSupply (on-chain token supply).

Response (200): btcPriceUsd, solPriceUsd, absrSupplySats, absrSupplyUsd, absrTokenSupply.

GET /api/arbitrage/transactions

ABSR-related transactions from the database (most recent first, up to 100). type is issue when a reserve SOL→BTC swap completes, or mint when the daily reserve sync mints ABSR to match the Bitcoin reserve balance. Requires DATABASE_URL; returns empty list if DB is not configured.

Response (200): { "transactions": [ { "signature", "txId", "type", "amountSats", "amountUsd", "blockTime" }, ... ] }. signature may be null for older issue rows; mint rows include the Solana tx signature.

5. Swap (reserve SOL → BTC only, via LI.FI)

Only SOL → BTC is supported. The server uses LI.FI to get a quote, sign and submit the Solana transaction, and track status. BTC is sent to the reserve Bitcoin address.

Custody: Swaps are fulfilled via a managed reserve (custodial). The server holds the reserve key and signs; users do not sign on-chain swap transactions. Funds are taken from the reserve Solana wallet; BTC is sent to the reserve Bitcoin address.

Confirmation time: Bitcoin settlement can take from several minutes up to a few hours depending on network congestion.

Success guarantee (swap): Once POST /api/swap/create returns initiated: true, the Solana side is submitted. BTC will be sent to the reserve address typically within a few minutes to an hour; in high congestion, allow up to a few hours. Poll GET /api/swap/status/:id until status is finished or failed.

GET /api/exchanges/transactions

Reserve SOL→BTC swap history (from Postgres).

Response (200): { "transactions": [ { "signature", "txId", "solAmount", "btcSats", "blockTime" }, ... ] }

GET /api/swap/min

Minimum SOL amount and current reserve SOL balance for SOL→BTC swap.

Response (200): { "minAmountSol", "balanceSol" }

GET /api/swap/estimate

Estimated BTC (sats) for a given SOL amount (from LI.FI quote).

Query: amountSol (positive number).

Response (200): { "amountSol", "estimatedBtcSats" }

POST /api/swap/create

Create a reserve SOL→BTC swap: server gets a LI.FI quote, signs the Solana transaction with the reserve key, and submits it. BTC is received at the reserve BTC address.

Body: { "amountSol": number } (positive).

Response (200): Returned immediately after the swap is initiated (Solana tx submitted). The agent receives the tx id (id / solanaSignature) and can disconnect; there is no need to wait for BTC confirmation (which can take minutes to hours). Poll GET /api/swap/status/:id only if completion status is needed. Fields: initiated: true, id, amountSol, expectedBtcSats, solanaSignature, status, statusUrl, message.

GET /api/swap/status/:id

Poll LI.FI transfer status. :id is the Solana transaction signature.

Response (200): { "id", "status", "amountSol", "btcSats" }. status: waiting, finished, or failed.

6. Invoices (agent token creation)

Agent submits token details → receives an invoice (fee in SOL). After paying the invoice to the treasury, the server creates the SPL token and adds it to the listing.

Success guarantee (token creation): After you call POST /api/invoices/confirm with a valid payment tx, the token is created and listed within a few seconds to under a minute (one Solana mint tx). Response includes token_id and mint_address.

POST /api/invoices

Create an invoice for token creation.

Body:

ParameterTypeRequired
namestringYes
symbolstringYes
decimalsnumber (0–9)No (default 9)
supplystringNo (default "0")
descriptionstringNo
creator_addressstringYes (SOL address to receive minted supply)
revoke_freeze_authoritybooleanNo
revoke_mint_authoritybooleanNo
revoke_update_authoritybooleanNo
metaplex_metadatabooleanNo

Response (200): invoice_id, treasury_address, fee_sol, fee_lamports, message, created_at.

GET /api/invoices/:id

Get invoice by ID (UUID).

Response (200): Full invoice fields including status (pending, completed), payment_tx_signature, token_id, paid_at, completed_at.

POST /api/invoices/confirm

Confirm payment: provide invoice ID and the Solana transaction signature that sent fee_sol to the treasury. On success, the server creates the token and lists it.

Body: { "invoice_id": string, "tx_signature": string } (or invoiceId, txSignature).

Response (200): { "ok": true, "status": "completed", "token_id", "mint_address", "message" }.

7. Tokens & listings

GET /api/tokens

List tokens. Optional query: creator (SOL address) to filter by creator.

Response (200): { "tokens": [ { "id", "name", "symbol", "decimals", "supply", "mint_address", "creator_address", ... }, ... ] }

POST /api/tokens

Create a token record in the database (metadata only; no on-chain mint). Body: name, symbol (required); decimals, supply, description, creator_address, mint_address, revoke_*, metaplex_metadata (optional). Returns the created token. Requires DATABASE_URL.

GET /api/tokens/:id

Get a single token by numeric ID.

GET /api/listings/search

Search listed tokens. Query: q (search string, name/symbol).

GET /api/listings

Get listing(s). Query: token_id (optional).

POST /api/listings/request

Request a listing (e.g. for an existing token). Body includes token_id and other fields; response returns fee and treasury address for payment.

POST /api/listings/confirm

Confirm listing payment (tx signature). On success the token is listed.

GET /api/listings/ensure-absr

Internal: ensure the ABSR token is registered in the tokens table and listed. Returns { "ok": true } or { "ok": false, "error": "..." }. Used by the server on startup and by listing search.

8. Explorer

GET /api/explorer/treasury

Returns treasury SOL address and optional metadata for the Token Explorer.

9. MCP server (agent tools)

An MCP server exposes the same API as tools so agents can discover and call them without hardcoding HTTP. Run npm run mcp (or node mcp-server.cjs) for stdio transport. Set API_BASE_URL to point at a different origin (default: https://www.solanaagent.app).

Tools: get_reserves, swap_min, swap_estimate, swap_create, swap_status, create_token_invoice, get_invoice, confirm_invoice, get_token. All take typed inputs and return JSON.

Cursor: Add to MCP settings (e.g. ~/.cursor/mcp.json) to use from Cursor: { "mcpServers": { "solana-agent-website": { "command": "node", "args": ["/path/to/website/mcp-server.cjs"], "env": { "API_BASE_URL": "https://www.solanaagent.app" } } } }

10. Errors

Errors are returned with appropriate HTTP status (400, 404, 502, 503) and a JSON body with error_code, error, and optional action (see deterministic flows section).
